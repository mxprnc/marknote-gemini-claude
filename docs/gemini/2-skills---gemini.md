## skills 란?
- "AI 에게 주는 업무 매뉴얼"
- 신입사원이 들어오면 매번 새로 생각해서 "이건 이렇게 하고, 저건 저렇게하고, 예를 들면 이렇게 ... " 라고 설명해야 하는 것들을 미리 정리해둔 것

skills 는 **agent skills** 라는 공개된 표준을 따른다. 즉 "범용 규격"이다. 따라서 Gemini CLI에서도 동일한 구조의 skills를 사용할 수 있다.

프롬프트 하나가 반복되고, 특정 작업에 대해 일관된 결과물을 얻고 싶을 때 skill로 정의해두면 된다. skill은 단순한 명령어가 아니라 **'Agent가 참고하는 지침서(SOP)'** 개념이다.

### skill 을 쓰는 것의 장점
- **재사용성**: 반복되는 복잡한 프롬프트를 한 번 정의해 계속 사용
- **일관성**: 모델이 정해진 절차와 규격에 따라 작업하도록 유도
- **이동성**: 프로젝트 간 또는 팀원 간에 쉽게 공유 가능

<br/>

### Gemini CLI 에서 지원되는지 여부
**Gemini CLI에도 Claude Code의 `skills`와 동일한 개념이 존재합니다.**

그 이유는 사용자께서 이미 문서에 적어주신 것처럼, 두 도구 모두 **'Agent Skills'**라는 개방형 표준(Open Standard) 규격을 따르기 때문입니다.

### 주요 특징
1.  **범용 규격**: `SKILL.md` 파일을 핵심으로 하는 폴더 구조는 특정 모델에 종속되지 않는 표준입니다.
2.  **이식성 (Portability)**: Claude용으로 만든 skill 폴더를 Gemini CLI의 skill 디렉토리(`~/.gemini/skills/` 등)로 복사하면 거의 그대로 작동합니다.
3.  **동작 방식**: 모델이 사용자의 요구사항을 분석하여 적절한 skill이 있다고 판단하면, 해당 `SKILL.md`에 기술된 지침(SOP)을 컨텍스트에 불러와서(Context Injection) 작업을 수행합니다.

따라서 작성 중이신 문서의 내용처럼 "AI 업무 매뉴얼"로서의 기능과 구조는 두 CLI 환경에서 동일하게 인터페이스됩니다.

<br/>

### 구조
![alt text](<./img/2-skills---gemini/CleanShot 2026-03-23 at 16.17.24@2x.png>)
<br/>

각 스킬은 전용 폴더 내의 `SKILL.md`를 진입점으로 하며, 필요한 경우 하위 폴더(`scripts/`, `resources/` 등)를 가질 수 있습니다.
이런 구조의 핵심은 **Lazy Loading**입니다. 모델은 세션 시작 시 모든 스킬 지침을 읽지 않고, 해당 스킬이 필요하다고 판단되는 시점에만 `SKILL.md`를 읽어들여 **컨텍스트 오염을 방지하고 토큰을 절약**합니다.

<br/>

### GEMINI.md vs Skills
| 항목 | GEMINI.md (프로젝트 지침) | **Skills (전문 업무 지침)** |
| :--- | :--- | :--- |
| **역할** | 프로젝트 전반의 기술 스텍, 코딩 컨벤션 정의 | 특정 태스크(예: 배포, 테스트 생성) 전문 수행 가이드 |
| **로드 시점** | 세션 시작 시 항상 로드 (Global) | **해당 태스크 발생 시에만** 로드 (Context-on-demand) |
| **구성 요소** | 텍스트 중심의 가이드라인 | 상세 절차(`SKILL.md`) + 자동화 스크립트(`scripts/`) |

<br/>

### MCP vs Sub-Agent vs Skills
- **MCP (Model Context Protocol)**: 외부 도구(GitHub, Slack, DB 등)와 물리적으로 연결하는 '커넥터'. 실행은 외부 서버에서 담당합니다.
- **Skills**: 프로젝트 내부에 포함된 **'업무 SOP + 실행 스크립트'**의 패키지. 모델이 스캔 후 직접 로컬 도구(`run_command` 등)를 써서 실행합니다.
- **Sub-Agent**: `browser_subagent`와 같이 특정 영역에 특화된 별도의 AI 인스턴스. 메인 에이전트가 복잡한 작업을 쪼개어 위임할 때 사용합니다.

<br/>

### SKILL.md 의 구조
```markdown
---
name: ppt-generator
description: "PPT 발표자료 자동 생성. 'PPT 만들어줘', '슬라이드 제작' 요청 시 활성화."
---
## 목적
사용자의 주제와 내용을 바탕으로 `scripts/` 폴더의 도구를 실행하여 PPT를 생성합니다.

## 작업 절차
1. 핵심 키워드 추출
2. `scripts/gen_ppt.py` 실행
3. 결과물 저장 경로 사용자 확인
```
- **Front-matter**: `name`과 `description`이 가장 중요합니다. Gemini는 이 설명을 보고 사용자의 요청과 일치하는 스킬을 찾아냅니다.
- **Body**: 모델이 따라야 할 단계별 지침, 제약 사항, 체크리스트를 기술합니다.

<br/>

## 스킬 만들기 (Gemini 스타일)
Gemini CLI는 별도의 UI나 플러그인(`skill-creator`)을 통하지 않고, **대화 중 Gemini에게 직접 생성을 요청**하는 것이 표준 워크플로우입니다.

### [방법] Gemini에게 생성을 지시하기
터미널에서 다음과 같이 자연어로 요청하세요:
> "특정 폴더의 마이그레이션 파일을 검수하는 스킬을 `.gemini/skills/migration-checker` 폴더에 만들어줘. `SKILL.md`와 필요한 체크용 파이썬 스크립트도 포함해줘."

**동작 원리:**
1. **의도 인식**: '스킬을 만들어줘' 또는 '가이드를 작성해줘'라는 요청을 받으면 Gemini가 스킬 생성 모드(Agentic Skill-building)로 진입합니다.
2. **파일 생성**: Gemini가 표준 규격에 맞는 `SKILL.md`와 보조 파일을 지정된 경로에 직접 생성합니다.
3. **즉시 적용**: 생성이 완료되면 별도의 재시작 없이 다음 질문부터 바로 해당 스킬이 작동합니다.

<br/>

## Skill 저장 위치 및 탐색 경로
Gemini CLI는 다음 세 위치를 순차적으로 탐색하여 스킬을 로드합니다.

| 구분 | 탐색 경로 (우선순위 순) | 비고 |
| :--- | :--- | :--- |
| **Workspace** | `.gemini/skills/` | 현재 프로젝트 전용 (Git 공유 가능, `.agent/` 경로와 혼용 가능) |
| **User** | `~/.gemini/skills/` | 내 로컬 PC의 모든 프로젝트에서 전역 사용 |
| **Extension** | `~/.gemini/extensions/<name>/skills/` | 설치된 외부 플러그인을 통해 공급되는 스킬 |

<br/>

## 기본 제공 (Bundled) 기능 대비
Claude Code의 고정 명령어들은 Gemini CLI에서 다음과 같은 유연한 기능으로 대체됩니다.

- **`/init`**: 프로젝트 분석 후 `GEMINI.md`를 생성하여 최적의 작업 효율을 설정합니다. (Claude의 `/simplify`와 유사)
- **자율 대량 변경**: 별도의 `/batch` 명령어 없이도, Gemini는 프롬프트만으로 다수의 파일을 병렬적으로 자동 수정하는 계획을 스스로 세워 실행합니다.
- **워크플로우 연동**: `.agent/workflows/` 내의 단계별 가이드를 읽어 복잡한 배포/테스트 자동화 절차를 수행합니다.
