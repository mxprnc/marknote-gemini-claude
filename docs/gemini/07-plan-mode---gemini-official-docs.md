## 1. 개요: 왜 계획이 필요한가?
대규모 리팩토링이나 복잡한 기능 추가 시, AI는 작업 도중 최초의 목표를 잊거나 엉뚱한 방향으로 진행될 수 있습니다. 이를 방지하기 위한 **작업 계획(Task Planning)**은 다음과 같은 이점을 제공합니다.

- **가시성**: 작업을 시작하기 전, 에이전트가 무엇을 하려는지 미리 검토할 수 있습니다.
- **집중력**: 에이전트가 현재 어떤 단계에 있는지 정확히 인지하여 문맥 손실을 방지합니다.
- **회복 탄력성**: 작업 중 오류 발생 시, 계획이 이정표 역할을 하여 올바른 궤도로 빠르게 복구합니다.

<br/>

## 2. 작업 계획(Task Planning)과 할 일 목록(Todo)
복잡한 작업을 요청할 때 "먼저 계획부터 세워줘"라고 명시하는 것이 가장 좋습니다.

### ① 계획 요청 및 검토
```
"이 프로젝트를 JavaScript에서 TypeScript로 마이그레이션하고 싶어. 먼저 계획부터 세워줘."
```
AI는 `write_todos` 도구를 사용해 할 일 목록을 생성합니다. 사용자는 "이 단계는 추가해줘", "순서를 바꿔줘"와 같이 실시간으로 피드백을 주어 계획을 수정할 수 있습니다.

### ② 실행 및 실시간 모니터링
계획이 확정되면 "좋아, 첫 단계부터 시작해줘"라고 명령합니다.
- **진행 상태 표시**: 하단 입력창 위에 현재 작업 단계(`[IN_PROGRESS]`)와 완료된 단계가 표시됩니다.
- **전체 확인 (`Ctrl+T`)**: 대화 내용이 길어져 목록이 가려져도, `Ctrl+T` 단축키로 언제든 전체 진행 상황을 열어볼 수 있습니다.

<br/>

## 3. Plan 모드 (Plan Mode) 상세 가이드
Plan 모드는 구현 전 아키텍처를 설계하고 전략을 수립하는 **읽기 전용(Read-only)** 안전 환경입니다.

### ① 진입 및 전환 방법
- **단축키**: `Shift+Tab`을 눌러 모드를 순환 선택할 수 있습니다.
- **명령어**: `/plan [목표]`를 입력하면 즉시 Plan 모드로 진입하며 리서치를 시작합니다.
- **환경 설정**: `/settings`에서 'Default Approval Mode'를 `Plan`으로 지정하면 항상 계획 모드로 시작합니다.

### ② 협업 계획 편집 (`Ctrl+X`)
AI가 제안한 계획이 마음에 들지 않거나 미세 조정이 필요할 때, 직접 수정하는 것이 가장 빠릅니다.
1. 계획서 검토 단계에서 **`Ctrl+X`**를 누릅니다.
2. 사용자의 외부 에디터(VS Code, Vim 등)가 열립니다.
3. 단계 삭제, 순서 변경, 의견(Comment) 등을 직접 입력하고 저장한 뒤 종료합니다.
4. AI는 수정 사항을 즉시 감지하여 전략을 다시 조정합니다.

### ③ 도구 제한 (Read-Only Safety)
Plan 모드에서는 시스템 보호를 위해 파일 쓰기나 삭제가 엄격히 제한됩니다. 오직 파일 읽기, 검색, 하위 에이전트를 통한 리서치 작업만 허용되어 예기치 못한 코드 수정을 방지합니다.

<br/>

## 4. 자동 모델 라우팅 (Automatic Model Routing)
품질과 속도의 균형을 위해 Gemini CLI는 현재 작업 페이즈에 따라 모델을 자동 전환합니다.

1. **계획 단계 (Planning Phase)**: 복잡한 추론과 설계가 필요하므로 고성능 **Pro 모델**로 자동 라우팅됩니다.
2. **구현 단계 (Implementation Phase)**: 계획 승인 후 실제 코드를 작성할 때는 빠른 응답을 위해 **Flash 모델**로 자동 전환됩니다.

<br/>

## 5. 고급 활용: 커스텀 및 최적화
- **저장 위치 변경**: 기본 임시 폴더 대신 프로젝트 내 `.gemini/plans/` 폴더에 계획서를 영구 저장하도록 `settings.json`에서 구성할 수 있습니다.
- **커스텀 후크(Hooks)**: 계획이 승인되거나 완료되었을 때 특정 외부 스크립트(예: 감사 로그 기록)를 실행하도록 설정할 수 있습니다.
