
## Q1 agent.md 파일의 claude, gemini 호환 여부 (2026/04/03)
agent.md 파일은 claude code, gemini cli 모두에서 통용되나요? 

### A
결론부터 말씀드리면, **`agent.md` 파일은 Claude Code의 전용 규격이며, 현재 Gemini CLI(또는 Google SDK 기반 툴)에서 공식적으로 통용되는 표준 파일은 아닙니다.**

하지만 두 도구가 지향하는 방향이 비슷하기 때문에, 개념적으로 어떻게 다른지 그리고 호환성을 어떻게 확보할 수 있는지 정리해 드릴게요.

---

#### 1. Claude Code와 `agent.md`
Anthropic에서 출시한 **Claude Code**는 프로젝트 루트 디렉토리에 있는 `agent.md` 파일을 자동으로 읽어 들입니다.

* **역할:** AI 에이전트가 프로젝트를 이해하기 위한 **'사용 설명서'** 혹은 **'페르소나 정의서'**입니다.
* **내용:** 코딩 스타일 가이드, 특정 라이브러리 사용법, 테스트 실행 방법, 프로젝트의 구조적 맥락 등을 기술합니다.
* **특징:** Claude Code는 이 파일을 발견하면 시스템 프롬프트에 자동으로 포함시켜, 매번 지시하지 않아도 컨텍스트를 유지합니다.

---

#### 2. Gemini CLI에서의 상황
Google의 **Gemini CLI**나 공식 SDK 기반 도구들은 현재 `agent.md`라는 특정 파일명을 자동으로 인식하는 표준 기능을 내장하고 있지 않습니다.

* **동작 방식:** 일반적으로 Gemini CLI는 사용자가 직접 입력하는 프롬프트나 `--system-instruction` 옵션을 통해 지침을 전달받습니다.
* **대안:** Gemini를 사용할 때 `agent.md`의 내용을 효과적으로 활용하려면, CLI 실행 시 해당 파일의 텍스트를 읽어서 시스템 프롬프트로 주입하는 별도의 쉘 스크립트를 사용해야 합니다.

---

#### 3. 요약 및 비교

| 비교 항목 | Claude Code | Gemini CLI |
| :--- | :--- | :--- |
| **기본 인식 파일** | `agent.md` | 없음 (표준화된 파일명 미지정) |
| **설정 방식** | 파일 존재 시 자동 로드 | 명령행 인자 혹은 수동 입력 |
| **커스텀 가능 여부** | 매우 높음 | 높음 (단, 수동 설정 필요) |

---

#### 💡 팁: 두 도구를 함께 사용하고 싶다면?
만약 프로젝트에서 두 AI를 번갈아 가며 사용하신다면, `agent.md`를 **프로젝트 지식 창고(Knowledge Base)**로 활용하는 것이 좋습니다.

1.  **Claude Code**를 쓸 때는 그대로 두면 자동으로 적용됩니다.
2.  **Gemini CLI**를 쓸 때는 아래와 같은 방식으로 파일을 참조하여 실행하세요. (예시 쉘 명령어)
    ```bash
    gemini ask "질문 내용" --system "$(cat agent.md)"
    ```

결국 `agent.md`는 **마크다운 형식의 텍스트 파일**일 뿐이므로, 파일명 자체보다는 그 안에 담긴 **프로젝트의 맥락(Context)**을 어떤 도구에 어떻게 주입하느냐가 핵심입니다.

이 외에 특정 CLI 도구(예: 오픈소스 커뮤니티에서 만든 Gemini 래퍼)를 사용 중이시라면, 해당 도구의 설정 파일(예: `.geminirc` 등)을 확인해 보시는 것을 권장합니다. 

혹시 현재 사용 중이신 특정 Gemini 관련 오픈소스 도구가 따로 있으신가요?

## Q2. GEMINI.md 파일의 프로젝트 내 재귀적인 적용 가능 여부
gemini cli 에서 GEMINI.md 는 프로젝트 디렉터리 모두에 대해 재귀적으로 적용되는 것은 아니죠?

### A
네, 정확하게 파악하셨습니다. **`GEMINI.md` 파일은 일반적으로 프로젝트의 루트(최상위) 디렉터리에 있는 것 하나만 인식하며, 하위 디렉터리마다 있는 파일들을 재귀적으로 수집하여 결합하지 않습니다.**

이는 성능과 보안, 그리고 컨텍스트 관리의 효율성 때문입니다. 구체적인 동작 원리와 주의사항은 다음과 같습니다.

---

#### 1. 작동 방식의 한계
대부분의 CLI 도구(Claude Code, Gemini 기반 커스텀 CLI 등)는 실행 시점에서 **현재 작업 디렉토리(CWD)** 또는 **Git 루트**를 기준으로 설정 파일을 찾습니다.

* **단일 파일 참조:** 실행 위치의 `GEMINI.md` 파일만 읽어 시스템 프롬프트에 주입합니다.
* **재귀적 병합 미지원:** 하위 폴더마다 `GEMINI.md`를 만들어 두어도, CLI가 알아서 그 파일들을 모두 찾아 내용을 하나로 합쳐주지는 않습니다. 

#### 2. 왜 재귀적으로 적용되지 않나요?
* **토큰 제한 (Token Limit):** 모든 하위 디렉터리의 지침을 합치다 보면 정작 중요한 코드나 질문을 입력할 '컨텍스트 창'이 부족해질 수 있습니다.
* **지침 충돌:** 루트의 지침과 하위 폴더의 지침이 서로 상충할 경우(예: "탭 사용" vs "스페이스 사용"), 어떤 것을 우선할지 결정하는 로직이 복잡해집니다.
* **성능 저하:** 실행할 때마다 파일 시스템을 전체 탐색하는 것은 응답 속도를 늦추는 원인이 됩니다.

---

#### 3. 해결 방법: 하위 디렉터리별 맥락이 중요하다면?

만약 특정 하위 모듈에 아주 특수한 규칙이 필요하다면 다음 전략을 고려해 보세요.

**A. 루트 `GEMINI.md`에 섹션 나누기**
하나의 파일 안에 디렉터리별 규칙을 명시하는 방법입니다.
```markdown
# Project Rules
- 기본적으로 TypeScript를 사용한다.

## /src/api (API 모듈 규칙)
- 모든 API 응답은 Result 타입을 래핑한다.

## /src/components (UI 모듈 규칙)
- Tailwind CSS만 사용한다.
```

**B. 실행 시점에 특정 파일 지정 (스크립트 활용)**
Gemini CLI를 실행할 때, 현재 작업 중인 폴더의 문서를 강제로 주입하는 쉘 별칭(Alias)이나 스크립트를 작성할 수 있습니다.
```bash
# 현재 폴더에 GEMINI.md가 있으면 그걸 쓰고, 없으면 루트걸 쓰는 방식 (사용자 설정 필요)
gemini ask --system "$(cat ./GEMINI.md || cat ~/project/GEMINI.md)"
```

---

#### 요약
`GEMINI.md`나 `agent.md` 같은 설정 파일은 **"프로젝트 전체를 관통하는 헌법"** 같은 역할을 하므로, 최상위에 하나만 두고 관리하는 것이 표준적인 사용법입니다. 만약 특정 폴더 내부에서만 돌아가는 독립적인 규칙이 필요하다면, 해당 폴더에서 직접 CLI를 실행하거나 루트 파일 내에 경로별 지침을 적어두는 것이 가장 효율적입니다.

혹시 특정 폴더마다 다른 규칙을 적용해야 하는 특별한 프로젝트 구조를 가지고 계신가요?
