## gstack
- github : https://github.com/garrytan/gstack
- star-history : https://www.star-history.com/garrytan/gstack#history

- **핵심 철학**: "1인 개발자가 마치 23명의 정예 팀원과 함께 일하는 것처럼" (Ship like a team).
- CEO, 개발리드, 디자이너, QA, 보안담당 등 23개의 전문 기술(skill)이 내장되어 역할 중심의 개발을 지원합니다.
- **제품 중심 접근**: 단순히 코드를 작성하는 "바이브 코딩"의 한계를 넘어, "Nobody wants it" 상황을 방지하기 위해 기획 단계부터 시장성, 수요 검증, 경쟁 분석을 철저히 수행합니다.
- **기술적 핵심 (Headless Chromium)**: 내장된 헤드리스 크로미엄 데몬을 통해 백그라운드에서 실시간으로 웹사이트를 탐색하고 테스트합니다. 쿠키, 로그인 세션, 탭 상태가 브라우저 레벨에서 유지되어 연속성 있는 작업이 가능합니다.
- **게리 탄 (Garry Tan)의 생산성**: Y Combinator의 게리 탄은 이 도구를 사용하여 50일 동안 주간 평균 1만 줄의 코드와 100개의 PR을 프로덕션 품질로 유지하며 그 효용성을 증명했습니다.
- **Proactive (능동적) 제안**: 사용자가 복잡한 명령어를 외울 필요 없이 "이게 왜 안 되지?"라고 물으면 `/investigate`를 제안하고, "배포하자"라고 하면 `/ship`을 실행하는 등 상황에 맞는 최적의 도구를 추천합니다.

### 🚀 gstack의 기술적 강점 (Bun + SQLite + HTTP Server)
gstack은 최신 런타임 기술을 활용하여 AI 개발 생산성을 극대화합니다.

1.  **Bun Runtime**: 
    - Node.js보다 수 배 빠른 실행 속도를 제공하는 **Bun**을 기반으로 구축되어, AI 에이전트의 복잡한 로직 처리를 지연 없이 수행합니다.
    - 패키지 매니저, 번들러, 유닛 테스트 Runner가 통합되어 있어 별도의 복잡한 설정 없이 즉시 실행 가능한 환경을 제공합니다.
2.  **Native SQLite (`bun:sqlite`)**:
    - 별도의 DB 서버 없이 **SQLite**를 사용하여 AI 에이전트의 상태, 브라우저 세션 데이터, 작업 히스토리를 로컬에 고속으로 저장하고 관리합니다.
    - 이를 통해 AI가 과거의 작업 맥락을 기억하고, 재시작 시에도 끊김 없는(stateful) 개발 경험을 제공합니다.
3.  **High-performance HTTP Server (`Bun.serve`)**:
    - Bun의 기본 내장 HTTP 서버를 활용하여 헤드리스 브라우저 데몬과 에이전트 간의 통신을 초저지연으로 처리합니다.
    - 가볍고 강력한 HTTP 통신 성능 덕분에 AI가 실시간으로 웹 스크린샷을 분석하거나 DOM을 조작하는 작업이 매우 매끄럽게 이루어집니다.

## superpowers 와 다른 점
superpowers
- "코드를 어떻게 쓸지" 에 집중
- 7단계 TDD 강제. (브레인스토밍, 계획, 테스트 먼저 작성, 구현, 리뷰)
- 테스트 없이 코드 금지 - 규율
gstack
- gstack 은 역할중심, superpower 는 과정 중심
- gstack 과 superpowers 는 경쟁개념이 아니라 공존가능, 보완하는 것

## 🛠️ 전체 워크플로우 요약
GStack은 **`Think → Plan → Build → Review → Test → Ship`** 순서로 스프린트가 진행되도록 설계되어 있습니다.

1.  **`/office-hours`**: 아이디어 검증 (할까 말까?)
2.  **`/plan-design-review`**: 설계 및 디자인 리뷰 (어떻게 만들까?)
3.  **`/review` & `/qa`**: 코드 리뷰 및 실제 구동 테스트 (잘 만들었나?)
4.  **`/ship`**: **최종 배포 및 PR 생성 (이제 내보내자!)**

> **요약하자면:** `/ship`은 "이제 다 됐으니 배포해!"라고 명령했을 때, AI가 **동기화, 테스트, 스타일 점검, PR 생성**까지 실수 없이 한 번에 처리해주는 **자동 출고 시스템**이라고 이해하시면 됩니다.

## /office-hours
- ceo 가 뭐 어디 직원한테 뭐 물어보는 거라고 했는데 이거 다른 영상에서 설명함
- e.g. "경쟁자는 있나요?", "왜 기존 서비스가 아니라 이걸 만들어야 하나요?"
- 이 과정을 거치면 이미 절반은 걸러진 상태에서 개발을 시작할수 있게 된다.(쓸데없는 제품을 안만들게 됨)
- gstack 을 쓸때 "세컨드 오피니언 받을까요?"하는 질문 역시 나오는데, 이건 다른 AI 모델에 질문을 하게 되는 걸 의미한다.

## /plan-design-review
- 디자인 계획 세우는 작업 역시 AI가 진행
- 실제 사이트를 들어가서 버튼 눌러보고 페이지 이동하고 입력도 해보고
- 디자인 리뷰 "너무 흔한 패턴입니다", UI/UX 분석 "CTA가 약합니다" 와 같은 AI의 리뷰들이 계속 생기는데, AI가 만든걸 AI가 비판하고, 고치고를 반복하는 과정
